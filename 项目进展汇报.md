# FDED_Eth 项目进展汇报

更新时间：2026-05-04

## 1. 项目目标

本项目目标是构建一个面向大模型 KVCache 的 FPGA 辅助内容寻址去重原型。当前工程从通用文件去重原型逐步扩展到 KVCache 场景，核心思路是：

```text
Host/CPU 负责切块、元数据、索引和调度
FPGA 负责指纹计算和热指纹表查询
GPU 负责热 KV page 的运行时驻留
```

当前重点围绕两个创新方向推进：

- 结构感知指纹分块：不再只把数据当普通字节流，而是按 KVCache 的 request、layer、K/V、head group、token range 等结构切成逻辑 KV page。
- 推理感知冷热调度：热 digest 放入 FPGA BRAM 热表，热 unique KV block 放入 GPU page pool，冷数据保留在 Host/文件存储侧。

## 2. 当前完成到哪一步

当前已经完成到：

```text
Host 文件/KV dump
  -> 结构化切块
  -> FPGA 计算 SHA-256 指纹
  -> Host sqlite 建索引
  -> unique block 存储
  -> KV page 级恢复
  -> GPU/Host/Cold 三层 KV block manager 原型
  -> PyTorch CUDA 后端真实分配 GPU 显存
```

更准确地说，当前已经具备一个 CPU + FPGA + GPU 显存分配的 KVCache 内容寻址去重原型，但还没有接入真实 LLM 推理框架，例如 vLLM、TensorRT-LLM 或 llama.cpp。

## 3. 系统架构

### 3.1 当前工程架构

```text
输入文件 / KVCache dump
        |
        v
FDED_host
  - fixed / FastCDC 切块
  - KV page 结构化切分
  - UDP 调用 FPGA
  - sqlite 指纹索引
  - unique block 写入
  - restore-file / restore-kv
  - KV block manager 调度
        |
        v
FDED_Eth FPGA
  - UDP 接收
  - SHA-256 / stream SHA-256
  - BRAM hot digest table
  - UDP 返回 digest / HOT_HIT
        |
        v
Result / data / write_dir
  - 结果表
  - sqlite 数据库
  - chunks/<digest>.bin
```

### 3.2 面向 KVCache 的逻辑架构

```text
Logical KV page
  request_id, model_id, layer_id, kv_kind, head_group, token_start
        |
        v
Unique KV block
  digest, ref_count, dtype, shape, length
        |
        v
Placement / Scheduling
  GPU_PAGE      热 KV block，当前支持 PyTorch CUDA tensor 真实显存分配
  HOST_CACHE    温数据缓存
  COLD_FILE     冷数据，chunks/<digest>.bin
        |
        v
FPGA hot digest table
  top-N 热 digest，用于减少 Host 查表压力
```

## 5. 已实现功能

### 5.1 FPGA 指纹计算与 Host 通信

已经实现：

- Host 通过 UDP 把数据发给 FPGA。
- FPGA 计算 SHA-256 digest。
- FPGA 通过 UDP 返回 digest。
- Host 用 `hashlib.sha256()` 做本地校验。
- 已处理早期一包延迟问题，加入 `seq_id` 协议进行请求响应匹配。

普通协议：

```text
request  = seq_id[4B] + payload
response = seq_id[4B] + status[1B] + digest[32B]
```

### 5.2 文件级去重闭环

Host 侧已经完成：

- fixed-size 切块。
- FastCDC 切块。
- FPGA hash。
- sqlite 指纹索引。
- unique chunk 写入 `chunks/<digest>.bin`。
- 文件恢复 `restore-file`。
- 结果导出为 Markdown 和 PNG。

文件恢复链路：

```text
file_runs
  -> chunk_events
  -> chunks/<digest>.bin
  -> restore-file
  -> SHA-256 校验一致
```

### 5.3 FPGA 热指纹表

已经实现 FPGA BRAM 热 digest 表。

Host 支持：

```powershell
python main.py clear-hot-table

python main.py load-hot-table --db-path data/fingerprints.db --limit 512
```

```powershell
python main.py process-file input.bin --load-hot-table --hot-limit 512 --hot-refresh-interval-s 10 --verify-hot-hit
```

统计字段包括：

```text

host_lookups
lookup_saved
hot_hit_ratio
hot_loaded
hot_refreshes
```

### 5.4 突破 1468B 逻辑长度限制

早期限制：

```text
1468B = 单个 UDP payload 的稳定传输上限
```

现在已经完成两种突破方式：

1. Host 分层指纹 `hierarchical`

```text
large chunk
  -> split into <=1468B fragments
  -> FPGA hash fragment
  -> Host 聚合 fragment digest
```

2. FPGA 原生跨包 `fpga-stream`

```text
large chunk
  -> 多个 UDP fragment
  -> FPGA 保持 SHA-256 中间状态
  -> END 包返回完整 chunk 的标准 SHA-256
```

当前更重要的是第二种，因为：

```text
fpga-stream digest == SHA256(完整原始数据)
```

### 5.5 KVCache 离线内容寻址原型

新增命令：

```powershell
python FDED_host\main.py process-kv-file <kv_dump.bin> --request-id req001 --model-id kv-demo --layer-id 0 --kv-kind K --head-group 0 --tokens-per-page 4 --bytes-per-token 1024 --digest-mode hierarchical --fragment-size 1468 --verify-local --print-pages
```

功能：

- 把二进制 KV dump 按 KV page 切分。
- 为每个 page 记录：
  - request_id
  - model_id
  - layer_id
  - kv_kind
  - head_group
  - token_start
  - token_count
  - dtype
  - shape
  - digest
- 重复 KV page 只保留一份 unique block。

恢复命令：

```powershell
python FDED_host\main.py restore-kv --request-id req001 --output-file Result\restored_kv_req001.bin
```

恢复链路：

```text
kv_pages
  -> unique_kv_blocks / fingerprints
  -> chunks/<digest>.bin
  -> restore-kv
  -> 原始 KV dump
```

### 5.6 KV block manager 与 GPU 显存后端

新增模块：

```text
FDED_host/src/fded_host/kv_block_manager.py
```

它现在是工程内的简化版 KV block manager，负责：

```text
Logical KV page
  -> Unique KV block
  -> GPU_PAGE / HOST_CACHE / COLD_FILE
```

支持两种 GPU 后端：

| 后端 | 说明 |
|---|---|
| `simulate` | 只模拟 GPU page pool，不真实占用显存 |
| `torch-cuda` | 使用 PyTorch `torch.empty(..., device='cuda:0')` 真实分配显存 |

运行示例：

```powershell
F:\anaconda3\envs\FDED\python.exe FDED_host\main.py simulate-kv-runtime --db-path data\fingerprints.db --request-id kv_req002 --repeat 2 --gpu-backend torch-cuda --gpu-pages 2 --gpu-page-bytes 1048576 --cache-pages 2 --fpga-hot-limit 1 --hot-refresh-interval 2 --policy inference-hot
```

验证结果中：

```text
gpu_backend         torch-cuda
gpu_pages           2
gpu_page_bytes      1048576
gpu_allocated_bytes 2097152
gpu_reserved_bytes  2097152
```

说明当前保留 2 个 GPU page，每页 1MiB，总共真实分配 2MiB CUDA 显存。

## 6. 数据库设计

当前 sqlite 数据库主要表：

| 表名 | 功能 |
|---|---|
| `fingerprints` | 通用 digest 索引，记录 digest、长度、首次出现位置、引用计数 |
| `chunk_events` | 文件级 chunk 事件 |
| `file_runs` | 文件处理 run 记录 |
| `kv_runs` | KV dump 处理 run 记录 |
| `kv_pages` | 逻辑 KV page 表 |
| `unique_kv_blocks` | unique KV block 表，记录 ref_count、hot_score、location 等 |
| `kv_access_events` | KV runtime 模拟访问事件 |

KVCache 逻辑映射关系：

```text
kv_pages.logical_page
  -> unique_block_id / digest
  -> unique_kv_blocks
  -> chunks/<digest>.bin 或 GPU_PAGE
```

- 接入 vLLM / TensorRT-LLM / llama.cpp。
- 修改真实 PagedAttention block table。
- 让 attention kernel 直接读取本工程管理的 GPU KV page。
- 实现真实推理过程中异步 hash、异步去重、异步 prefetch。

   - FPGA stream hash 性能。
   - GPU/CPU/FPGA 分层调度实验。
   - 与纯 CPU hash、无 FPGA 热表、无 GPU prefetch 的 baseline 对比。

## 12. 当前阶段总结

当前项目已经从“FPGA 计算 SHA-256 的以太网 demo”推进到了一个较完整的内容寻址 KVCache 原型：

- FPGA 端：已完成 SHA-256、跨包 stream hash、热 digest 表。
- Host 端：已完成切块、索引、去重、恢复、结果导出。
- KVCache 层：已完成结构化 KV page、unique block、逻辑映射、重复复用、恢复验证。
- 调度层：已完成 GPU/Host/Cold 三层 KV block manager。
- GPU 层：已支持 PyTorch CUDA 后端真实分配显存。

当前最主要的缺口是还没有接入真实 LLM 推理框架。下一阶段应从真实 KVCache trace 和 vLLM/PagedAttention block manager 集成入手，把当前原型从离线/模拟进一步推进到在线推理系统。
