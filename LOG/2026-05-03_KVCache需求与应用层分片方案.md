# 2026-05-03 KVCache 需求与应用层分片方案

## 主题

结合 `新的要求.txt`，讨论当前 FDED_Eth 工程如何从文件去重原型推进到“基于 FPGA 的内容寻址 KVCache”，并重点分析 `1468B` UDP payload 限制是否需要去掉，以及能否通过 Host 应用层分片减少 FPGA 端改动。

## 当前工程状态判断

- 当前工程已经完成：
  - Host 侧切块：固定大小切块与 FastCDC。
  - FPGA 侧 SHA-256 计算。
  - Host 与 FPGA UDP 通信。
  - Host sqlite 指纹索引与唯一 chunk 写入。
  - FPGA BRAM 热 digest 表。
  - Host 加载热表、统计 `HOT_HIT/MISS`、跳过部分 Host 查表。
  - `restore-file` 可根据 chunk 元数据恢复文件。
- 当前系统更准确的定位是：

```text
Host-side chunking
+ FPGA-side hashing
+ Host-side fingerprint indexing
+ FPGA hot digest table prototype
```

- 距离新的目标“基于 FPGA 的内容寻址 KVCache”还缺：
  - KVCache 结构感知 page/chunk 定义。
  - `request/layer/head/token` 到 unique KV block 的逻辑映射。
  - GPU/CPU/FPGA 分工设计。
  - 不阻塞 LLM attention 正常访问的后台去重路径。
  - KV page 恢复/重定向机制。

## 对新需求中三个问题的结论

### 1. 去重切多大块

- 当前工程受 UDP payload 限制，稳定 chunk 上限约为 `1468B`。
- 当前文件去重实验可继续使用：

```powershell
python -B main.py process-file <input> --chunk-mode cdc --min-size 512 --avg-size 1024 --max-size 1468
```

- 面向 KVCache，不建议用纯字节流 CDC 作为主方案。
- 推荐使用结构感知 KV page：

```text
chunk = model_id + layer_id + head/group + token block + dtype/shape + KV bytes
```

- 推荐实验参数：

```text
8 / 16 / 32 / 64 tokens per KV page
```

- 起步建议用 `16 tokens/page` 或 `32 tokens/page`，再比较重复率、索引开销、恢复延迟和吞吐损失。

### 2. FPGA 去重放在哪里

- FPGA 不应放在 LLM attention 同步读路径上。
- 推荐放在 KVCache 写入后的异步后台路径：

```text
GPU 正常生成/读取 KVCache
  -> CPU/KVCache Manager 记录逻辑 page
  -> 后台将 KV page 或 page fragment 发给 FPGA 计算指纹
  -> CPU 更新冷/热索引与逻辑映射
```

- 三类设备分工：

```text
GPU:
  - 执行 prefill/decode/attention
  - 保留当前需要的热 KV page
  - 不等待 FPGA 判重

CPU:
  - KV page 元数据管理
  - 逻辑页到 unique block 的映射
  - 冷索引、引用计数、回收
  - 调度 FPGA 热表加载和刷新

FPGA:
  - 计算 SHA-256 或后续快速 fingerprint
  - 查询 BRAM 热 digest 表
  - 返回 MISS/HOT_HIT
  - 后续可扩展 Bloom filter、组相联热表、PCIe 通道
```

### 3. 去重后如何访问 unique 数据

- 不修改大模型本身，让 LLM 仍访问逻辑 KV page。
- 在 KVCache Manager 层增加 indirection：

```text
LogicalKVPageTable:
  request_id, layer_id, head_group, token_start
    -> unique_block_id, offset, length

UniqueKVBlockTable:
  unique_block_id
    -> digest, physical_location, shape, dtype, ref_count
```

- 如果 unique block 已在 GPU KV pool，直接返回 GPU address。
- 如果 unique block 在 CPU DRAM/SSD，先 restore/prefetch 到 GPU page pool，再返回给运行时。
- 当前 `restore-file` 的思想可迁移为 KV page 恢复：

```text
逻辑 page 序列 -> digest/unique_block_id -> unique KV block -> 恢复到 GPU KV page
```

## 关于 `1468B` 限制

### 是否要去掉

- 是，最终需要突破 `1468B` chunk 限制。
- 但不应表述为“真正不限长度 hash”，更准确是：

```text
支持跨 UDP 包的大 chunk / KV page 指纹计算。
```

- SHA-256 算法本身支持任意长度，当前限制来自：
  - UDP 单包 payload。
  - FPGA 当前按“一包一个 chunk”读取 RX RAM。
  - Host 当前 `max_data_len` 默认 `1468`。

### 直接改 FPGA 的路线

可设计跨包协议：

```text
chunk_id
chunk_total_len
fragment_offset
fragment_len
flags: START / MIDDLE / END
payload fragment
```

FPGA 端有两种实现：

- 跨包缓存后再 hash：实现相对简单，但需要 BRAM/DDR 缓存完整 chunk。
- 边收边流式 hash：最终更合理，但要大改 FPGA 状态机、包序处理、padding、错误恢复。

短期不建议立即大改 FPGA。

## 推荐的低改动方案：Host 应用层分片

### 核心思路

在 Host 侧把大 chunk 或 KV page 分成多个 `<=1468B` fragment，继续调用当前 FPGA `hash_chunk()`，FPGA 仍然只负责小片段 SHA-256。

流程：

```text
large chunk / KV page
  -> Host split into fragment_0..fragment_n
  -> FPGA hash each fragment
  -> Host collect fragment_digest[]
  -> Host aggregate final chunk_digest
  -> sqlite fingerprint index
```

### 聚合指纹

最终 chunk digest 不再等于原始：

```text
SHA256(large_chunk_bytes)
```

而是分层指纹：

```text
chunk_digest = SHA256(
    "FDED_CHUNK_V1"
    + chunk_total_len
    + fragment_size
    + fragment_count
    + digest_0
    + digest_1
    + ...
)
```

这属于 Merkle-style / hierarchical fingerprint，只要所有数据都使用同一规则，就可以用于去重。

### 优点

- FPGA 端基本不用改。
- 可以突破当前 `1468B` chunk 限制。
- 可快速支持 `4KB / 8KB / 16KB / 64KB` KV page 实验。
- 与“结构感知分块”创新点兼容。
- 后续 FPGA 改成真正流式 hash 时，Host 只需要切换 digest mode。

### 缺点

- 聚合 digest 不是标准 `SHA256(完整 chunk 原文)`。
- 大 chunk 需要多次 UDP 往返，当前以太网仍是瓶颈。
- Host 需要维护 fragment 级统计与 chunk 级最终 digest 的对应关系。

## 建议下一步工程修改

### Host 侧优先改动

- 在 `FDED_host/src/fded_host/pipeline.py` 增加 `hash_large_chunk()`：

```text
if len(chunk.data) <= max_data_len:
    reply = client.hash_chunk(seq_id, chunk.data)
    chunk_digest = reply.digest
else:
    fragments = split(chunk.data, max_data_len)
    fragment_digests = [client.hash_chunk(fragment_seq, fragment).digest for fragment in fragments]
    chunk_digest = aggregate_fragment_digests(...)
```

- 在统计中增加：

```text
fragment_hashes
large_chunks
avg_fragments_per_large_chunk
digest_mode = raw_sha256 | hierarchical_v1
```

- 在 CLI 增加可选参数：

```text
--fragment-size 1468
--digest-mode hierarchical
--max-chunk-size 4096/8192/16384
```

### FPGA 侧暂不大改

- 保持当前 `seq_id + payload -> status + digest` 协议。
- 保持当前 BRAM 热 digest 表。
- 热表中存放最终 chunk digest，而不是 fragment digest。
- 因为最终 chunk digest 在 Host 聚合产生，热表加载仍走 sqlite top-N。

注意：如果热表查询仍发生在 FPGA 每个 fragment hash 之后，那么 FPGA 只能对 fragment digest 查热表。短期建议：

```text
fragment digest 只用于 Host 聚合；
chunk-level HOT_HIT 仍由 Host sqlite 或后续 Host-loaded final digest 热表机制处理。
```

若要 FPGA 直接对大 chunk final digest 查热表，则仍需要 FPGA 支持聚合或流式 final digest。

## 推荐实验路线

1. 先保留当前 `1468B` 原型，完成文件去重闭环实验：

```text
process-file/process-dir -> unique chunk write -> restore-file -> 校验一致
```

2. 增加 Host 应用层分片与 hierarchical fingerprint，支持大 chunk：

```text
4KB / 8KB / 16KB / 64KB
```

3. 构造 KVCache 离线模拟数据：

```text
model_id, layer_id, head_group, token_start, token_count, dtype, shape, data
```

4. 对比实验：

```text
1468B CDC
4KB fixed/page
8KB fixed/page
16 tokens/page

```

5. 再考虑 FPGA 真正流式 hash 或 PCIe 通信优化。

## 当前决策

- 不立即大改 FPGA 做跨包流式 SHA-256。
- 先采用 Host 应用层分片 + FPGA fragment hash + Host hierarchical aggregation。
- 论文表述上可称为“分层内容指纹”或“hierarchical fingerprinting”。
- KVCache 创新重点继续放在：
  - 结构感知指纹分块。
  - 推理感知冷热索引调度。
  - GPU/CPU/FPGA 分工。
  - 内容寻址 KVCache 的逻辑重定向。

## 16:18 1468B 逻辑长度限制突破记录

### 结论

- 当前已经突破“一个逻辑 chunk 最大 1468B”的限制。
- 现在的含义变为：

```text
1468B = 单次 UDP fragment 传输上限
逻辑 chunk/page 大小 = Host fixed/CDC/KV page 决定，可大于 1468B

  -> FPGA hashes each fragment
  -> Host aggregates fragment digests
  -> one logical chunk digest
```

```text
SHA256("FDED_CHUNK_V1" + chunk_length + fragment_size + fragment_count + fragment_digest...)
```
