# 2026-04-27 SHA256 多块版本 1468B 验证通过记录

## 1. 背景

此前工程已经完成：

- 请求协议：`4B seq_id + payload`
- 响应协议：`4B seq_id + 32B digest`
- 单块 SHA-256 版本联调通过

本次目标是继续将 FPGA 侧 SHA-256 从“单块版本”扩展为“多块版本”，使其能够对超过 `51B` 的真实业务数据继续正确计算 SHA-256。

## 2. 本次修改目标

本次扩展的核心目标是：

- 保留现有 UDP 收发链路
- 保留 `seq_id + digest` 协议
- 将 `udp_sha256_oneblock.v` 从“单块拼包后一次计算”改为“多块顺序送入 SHA256 核”
- 在不改接收 RAM IP 容量的前提下，先验证多块 SHA-256 逻辑本身是否正确

## 3. 关键实现结论

当前版本中：

- `sha256_core / sha256_stream` 本身支持多块串接
- 真正需要扩展的是前端分块与 padding 逻辑
- `udp_sha256_oneblock.v` 已改为：
  - 先从接收 RAM 中提取前 4B `seq_id`
  - 再按 `64B` block 顺序读取真实 payload
  - 对最后一块自动生成 SHA-256 标准 padding
  - 通过 `s_tlast` 标记最终块

因此当前 SHA-256 计算路径已经从：

- `单块 payload -> 单块 padding -> SHA`

扩展为：

- `任意多块 payload -> 分块送入 -> 末块 padding -> SHA`

## 4. 过程中出现的现象

首次扩展后，在测试较长数据时曾出现两类问题：

- 中等长度数据返回 `seq=0`、`digest=0`
- 更长数据直接超时

进一步分析后确认：

### 4.1 全零回包

问题不一定是主机测试错误，而是 `mac_test.v` 中错误回包路径会把整包 `36B` 全部写成 `0`，导致看起来像：

- `r-seq = 0`
- `digest = 0`

为便于定位，后续已调整为：

- 即使 SHA 路径报错，也保留真实 `seq_id`
- 仅将 `digest` 置零

### 4.2 超长 UDP 包超时

进一步确认：

- FPGA 接收 RAM 容量允许缓存更大数据
- 但当前普通以太网 UDP 链路未做 IP 分片重组

在标准 MTU `1500B` 下：

- `UDP payload` 最大约为 `1472B`
- 由于协议前 4B 用于 `seq_id`
- 当前“真实 hash 数据”的网络实测上限应按 `1468B` 理解

因此：

- `2044B` 是接收 RAM 理论上可容纳的真实数据上限之一部分推导结果
- 但在当前未支持 IP 分片重组的网络链路中，不应直接拿它做普通 UDP 联调目标
- 当前合理、可直接验证的上限应是 `1468B`

## 9. 当前上限的准确表述

当前应区分两个“上限”概念：

### 9.1 FPGA 缓存/架构上限

受当前接收 RAM 容量影响，FPGA 内部缓存能力高于单块版本，理论上还可继续向更大输入推进。

### 9.2 当前网络联调实测上限

在当前普通 UDP 测试方式、未支持 IP 分片重组的前提下：

- 当前真实业务数据长度的稳定实测上限为 `1468B`

因此后续如果希望继续验证或支持更长输入，需要进一步选择以下方向之一：

- 增大并重构接收路径，支持更大的完整应用层数据组织
- 在主机与 FPGA 之间实现应用层分片协议
- 改成真正流式 SHA-256 接收处理，而不是依赖“整包先存入 RAM 再计算”
