## 11. SHA256 联调修复与验证通过

在用户重新生成 bit 并联调后，出现过一次：

- UDP 可以正常收发
- FPGA 可以返回 32B digest
- 但 `abc` 的返回值与主机 `hashlib.sha256(b"abc")` 不一致

进一步定位确认：

- 返回的错误 digest 实际对应字符串 `aab`
- 说明链路和 SHA256 核都在工作
- 问题出在 `udp_sha256_oneblock.v` 读取 `udp_rx` RAM 的时序

根因：

- `udp_rx` RAM 为同步读口
- 地址切换后需要等待一拍数据稳定
- 原状态机在连续读取 payload 时少等了一拍，导致读取错位

修复内容：

- 修改 `FDED_Eth/FDED_Eth.srcs/sources_1/new/eth_src/mac/udp_sha256_oneblock.v`
- 新增 `ST_READ_DONE`
- 在连续读 RAM 时，每次地址递增后重新经过 `ST_READ_SETUP`
- 保证每个 payload 字节都按同步 RAM 时序正确采样

修复后重新生成 bit 并测试：

- 发送：`abc`
- FPGA 返回：
  - `ba7816bf8f01cfea414140de5dae2223b00361a396177a9cb410ff61f20015ad`
- 与主机软件计算结果一致

结论：

- 当前版本已经成功实现
  - `主机 UDP payload -> FPGA SHA-256 -> FPGA 返回 32B digest`
- 当前约束仍为：
  - `payload <= 55B`
  - 单块 SHA-256

## 14. 测试脚本补偿方案

为了继续联调与验证，测试脚本增加了可选参数：

- `--pipeline-lag 1`

含义：

- 主机测试端按“FPGA 返回前一包 digest”这一行为进行补偿验证

脚本处理逻辑包括：

- 发送前尽量清空 socket 中残留的旧 UDP 包
- 在 `lag=1` 模式下，按“当前发送，下一次接收对应当前 digest”的规则比较结果

这样做的目的：

- 先验证 FPGA 计算结果本身是否正确
- 在正式修复协议前，允许继续做稳定联调

## 15. 当前工程建议

本次对话最终明确：

- **测试侧补偿可以作为临时联调方案**
- **不建议把这种行为直接当正式工程方案**

原因：

- 正式系统中，“第 n 包请求返回第 n-1 包 digest”会使主机协议复杂化
- 丢包、重发、乱序时很难可靠匹配
- 后续做分块、批量 hash、去重索引时会明显增加管理复杂度

更合理的正式方案有两类：

- 方案 1：修 FPGA，使其恢复为严格“一次请求对应一次响应”
- 方案 2：把系统定义为流水线协议，并在请求/响应中显式加入 `seq_id`

当前建议优先级：

- 联调阶段：可继续使用 `--pipeline-lag 1`
- 正式实现阶段：应改为“同步请求-响应”或“带序号的流水线协议”

## 16. seq_id 流水线协议落地与修复完成

在后续工作中，决定不再停留在测试侧补偿，而是正式将协议升级为：

- 请求：`4B seq_id + payload`
- 响应：`4B seq_id + 32B digest`

这样做的目的是：

- 让主机和 FPGA 在流水线场景下仍能明确区分“这条 digest 属于哪一笔请求”

### 16.1 初始改造后的现象

协议改造后，单次请求曾出现过：

- 有时第一次返回的 digest 错误
- 随后再次发送同样请求则恢复正确

多用例连续测试中则稳定表现为：

- `seq_id` 是当前包
- `digest` 是上一包

也就是：

- `response(seq_n, hash(payload_{n-1}))`

说明：

- `seq_id` 通路已经接通
- 但 `digest` 结果没有和同一笔事务严格绑定

### 16.2 根因定位

最终定位到：

- `udp_sha256_oneblock.v` 中对 `sha_digest_valid` 的使用方式存在问题
- 模块在等待新一笔 digest 时，可能再次采到上一笔仍保持有效的 `digest_valid`
- 从而把旧 digest 当作当前事务结果锁存

这正是“当前 `seq_id` 搭配上一包 `digest`”的根本原因

### 16.3 关键修复

本次正式修复包括：

- `udp_sha256_oneblock.v`
  - 增加 `seq_id` 输出
  - 从前 4B 请求头提取 `seq_id`
  - 仅对后续真实 payload 计算 SHA-256
  - 增加 `sha_digest_valid_d0`
  - 只在 `sha_digest_valid` 上升沿锁存 digest

- `mac_test.v`
  - 回包格式改为 `4B seq_id + 32B digest`
  - 在接收侧触发 SHA 和锁存结果
  - 通过 `toggle + ack` 将整笔结果跨时钟域交给发送侧
  - 发送侧只发送已经绑定好的 `seq_id + digest`

- `FDED_test/test_udp_sha256.py`
  - 改为按 `seq_id` 协议发包/收包
  - 同时校验返回 `seq_id` 与 `digest`
  - 增加多用例测试集
