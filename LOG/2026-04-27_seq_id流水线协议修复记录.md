# 2026-04-27 seq_id 流水线协议修复记录

## 1. 背景

在原有 `UDP payload -> FPGA -> SHA-256 -> UDP reply` 基础上，为了支持后续流水线式请求/响应对齐，对协议做了升级：

- 请求格式改为：`4B seq_id + payload`
- 响应格式改为：`4B seq_id + 32B digest`

目的：

- 即使系统内部存在固定流水延迟，也能通过 `seq_id` 明确请求和响应的对应关系

## 2. 第一阶段现象

单次请求测试时：

- `seq_id` 和 `digest` 都能正确返回

但多用例连续测试时出现：

- `seq_id` 是当前包
- `digest` 是上一包的 hash

表现形式为：

- `response(seq_n, hash(payload_{n-1}))`

说明：

- `seq_id` 通路已经接通
- `digest` 结果没有和同一笔事务严格绑定

## 3. 主要定位过程

### 3.1 初步怀疑：跨时钟绑定不完整

在 `mac_test.v` 中，最初已尝试：

- 在接收侧做 SHA
- 在发送侧回包
- 用 `toggle` 方式把结果从接收时钟域送到发送时钟域

但实测发现：

- `seq_id` 能对上
- `digest` 仍然落后一包

因此判断：

- 单纯把 `seq_id`、`digest` 暂存并跨域还不够
- 还需要进一步确认 SHA 结果“何时真正更新”

### 3.2 最终根因：`sha_digest_valid` 不是单拍边沿使用

进一步分析后确认：

- `udp_sha256_oneblock.v` 在 `ST_WAIT_DIGEST` 中，直接用 `sha_digest_valid` 作为“结果到达”条件
- 若 `sha_digest_valid` 在前一笔结果后保持为高，下一笔进入等待状态时，会再次看到这个高电平
- 导致模块在新一笔事务中错误地锁存了上一笔的 `digest`

这正对应实际联调现象：

- `seq_id` 已经是新的
- `digest` 仍然是旧的

## 4. 本次关键修改

### 4.1 协议升级

修改内容：

- 请求从“纯 payload”改为 `4B seq_id + payload`
- 响应从“32B digest”改为 `4B seq_id + 32B digest`

影响：

- 原先最大单块数据长度为 `55B`
- 由于前 4B 留给 `seq_id`
- 当前真实业务数据最大长度变为 `51B`

### 4.2 `udp_sha256_oneblock.v`

文件：

- `FDED_Eth/FDED_Eth.srcs/sources_1/new/eth_src/mac/udp_sha256_oneblock.v`

修改点：

- 增加 `seq_id` 输出
- 从接收 RAM 前 4B 解析 `seq_id`
- 只对后续真实 `payload` 做 SHA-256
- 新增 `sha_digest_valid_d0`
- 在 `ST_WAIT_DIGEST` 中只对 `sha_digest_valid` 上升沿响应：
  - `if (sha_digest_valid && ~sha_digest_valid_d0)`

这一步是最终修复多用例错位的关键。

### 4.3 `mac_test.v`

文件：

- `FDED_Eth/FDED_Eth.srcs/sources_1/new/eth_src/mac/mac_test.v`

修改点：

- 回包长度改为 `36B`
- 回包内容改为：
  - 前 4B：`seq_id`
  - 后 32B：`digest`
- 在接收侧做 SHA 触发和结果锁存
- 用 `toggle + ack` 方式，把整笔结果从接收时钟域可靠交给发送时钟域
- 发送侧只发送已经绑定好的 `seq_id + digest`

### 4.4 Python 测试脚本

文件：

- `FDED_test/test_udp_sha256.py`

修改点：

- 按新协议发包：`seq_id + payload`
- 按新协议解包：`seq_id + digest`
- 测试时同时校验：
  - 返回 `seq_id`
  - 返回 `digest`
- 增加多用例测试集，覆盖：
  - `abc`
  - 单字节 `0x00`
  - 空串
  - `hello`
  - ASCII 文本
  - 二进制字节序列
  - `51B` 边界值

## 5. 最终验证结果

在重新综合、实现、生成 bit 并重新下载后，进行了两类验证。

### 5.1 单次验证

测试：

- `--message abc`

结果：

- `seq_id` 正确
- `digest` 正确
- 测试通过

### 5.2 多用例验证

测试：

- `--suite`

结果：

- `7/7 passed`

具体通过用例：

- `abc`
- `single_zero`
- `empty`
- `hello`
- `ascii_sentence`
- `binary_00_07`
- `len_51_A`

说明当前版本已经确认实现：

- `request(seq_id + payload) -> FPGA SHA-256 -> response(seq_id + digest)`

并且：

- 单次请求响应正确
- 多次连续请求响应正确
- `seq_id` 与 `digest` 已经严格绑定为同一笔事务

## 6. 当前结论

截至本次修复完成，当前系统已验证通过：

- UDP 主机发送带 `seq_id` 的请求
- FPGA 正确提取 `seq_id`
- FPGA 对真实 payload 做 SHA-256
- FPGA 返回带相同 `seq_id` 的 digest
- 多用例连续发送下也能稳定对齐

当前实现仍然属于：

- 单块 SHA-256 版本
- 真实数据长度需满足 `<= 51B`

后续若扩展到 CDC 块级处理，需要进一步升级为：

- 多块流式 SHA-256
- 支持任意长度块输入
