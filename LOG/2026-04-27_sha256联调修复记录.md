# 2026-04-27 SHA256 联调修复记录

## 1. 现象

主机测试脚本：

- `C:\Users\14854\Desktop\Eth_test\test_udp_sha256.py`

发送：

- payload: `616263`
- 明文: `abc`

主机期望的 SHA-256：

- `ba7816bf8f01cfea414140de5dae2223b00361a396177a9cb410ff61f20015ad`

FPGA 首次返回：

- `38760eabb666e8e61ee628a17c4090cc50728e095ff24218119d51bd22475363`

表现为：

- UDP 收发链路正常
- FPGA 能稳定返回 32B digest
- 但 digest 与主机软件计算结果不一致

## 2. 定位过程

对返回 digest 做反推后确认：

- `38760eabb666e8e61ee628a17c4090cc50728e095ff24218119d51bd22475363`
- 实际上对应字符串 `aab` 的 SHA-256

因此可以判断：

- 问题不在 UDP 通信链路
- 问题不在 SHA256 算法核心
- 问题出在 FPGA 从 `udp_rx` RAM 读取 payload 时发生了错位

即：

- 主机发的是 `abc`
- FPGA 实际参与 hash 的内容变成了 `aab`

## 3. 根因分析

问题文件：

- `FDED_Eth/FDED_Eth.srcs/sources_1/new/eth_src/mac/udp_sha256_oneblock.v`

根因是：

- `udp_rx` 使用的是同步 RAM 读口
- 读地址切换后，数据不是同拍立即有效，而是需要等待一拍

旧状态机在连续读取 payload 时：

- 地址递增后没有为 RAM 输出稳定留出足够的等待周期
- 导致上一地址的数据被重复采样

结果就是：

- 第一个字节 `a` 被读了两次
- 最后一个字节 `c` 没有正确进入 hash 输入

最终输入从 `abc` 变成了 `aab`

## 4. 修改内容

修改文件：

- `FDED_Eth/FDED_Eth.srcs/sources_1/new/eth_src/mac/udp_sha256_oneblock.v`

本次修复采用最小改动：

- 新增状态 `ST_READ_DONE`
- 在 `ST_READ_DATA` 中，每次地址加一后重新回到 `ST_READ_SETUP`
- 让同步 RAM 在每次地址切换后都有一拍准备时间
- 最后一个字节采完后先进入 `ST_READ_DONE`，再进入 `ST_BUILD_BLOCK`

修复目的：

- 保证写入 `payload_buf[]` 的每个字节都与 `ram_read_addr` 对应
- 保证 SHA256 输入块按真实 UDP payload 顺序拼接

## 5. 验证结果

重新生成 bit 并下板后，再次运行测试脚本：

- payload: `616263`
- 明文: `abc`

主机期望：

- `ba7816bf8f01cfea414140de5dae2223b00361a396177a9cb410ff61f20015ad`

FPGA 返回：

- `ba7816bf8f01cfea414140de5dae2223b00361a396177a9cb410ff61f20015ad`

结果：

- `result pass: digest matched`
- `summary 1/1 passed`

说明当前链路已经实现：

- 主机发送 UDP payload
- FPGA 读取 payload
- FPGA 计算单块 SHA-256
- FPGA 返回 32 字节 digest

## 6. 当前结论

当前版本已经验证通过以下目标：

- `UDP payload -> FPGA -> SHA-256 -> UDP reply digest`

当前已确认的约束：

- 仅支持单块 SHA-256
- payload 长度需满足 `<= 55B`
- 返回内容固定为 `32B digest`

后续如果继续扩展，可考虑：

- 支持大于 55B 的多块 SHA-256
- 对超长 payload 返回明确错误码或错误包
- 增加更系统的联调测试样例，例如空串、55B 边界值、二进制 payload
