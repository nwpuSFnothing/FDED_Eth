# 2026-05-05 KVCache 去重运行时设计讨论总结

## 23:22 讨论主题

### 用户需求

- 梳理当前项目中 FPGA、CPU、GPU 的真实功能分工。
- 明确当前项目与真实 LLM runtime 中 KVCache 访问路径的区别。
- 判断当前项目是否本质上属于 KVCache 压缩/去重。
- 明确“多个请求共享同一个 KV page”时，访问单个请求应该如何理解。

### 当前项目定位

- 当前项目是一个 FPGA 辅助的 KVCache 内容寻址去重原型。
- 已经真实完成的链路是：

```text
离线 KVCache dump / 文件数据
  -> CPU 切块或切 KV page
  -> FPGA 计算 SHA-256 digest
  -> CPU 用 sqlite 建立 digest 索引
  -> 重复块复用已有 unique block
  -> 唯一块写入 chunks/<digest>.bin
  -> 可恢复原始文件或 KV dump
```

- 当前 GPU/Host/Cold 调度部分仍是运行时模拟/原型：

```text
kv_pages 访问 trace
  -> CPU 查 unique_kv_block
  -> Python KV block manager 判断 GPU_PAGE / HOST_CACHE / COLD_FILE
  -> 模拟 hit、miss、eviction、hot_score
  -> 可选 torch-cuda 后端真实分配 CUDA 显存
```

- 当前还没有接入 vLLM、TensorRT-LLM、llama.cpp 等真实 LLM runtime。
- 当前还没有实现真实推理时多个 logical KV page 共享同一个 physical GPU KV block。

### FPGA / CPU / GPU 分工

- FPGA 当前真实负责：
  - UDP 收发。
  - SHA-256 指纹计算。
  - 跨 UDP 包 stream hash。
  - BRAM hot digest table 查询。
  - 返回 `seq_id + status + digest`。

- CPU/Host 当前真实负责：
  - 文件或 KV dump 读取。
  - fixed/FastCDC 切块。
  - KV page 结构化切分。
  - 通过 UDP 调用 FPGA。
  - sqlite 指纹索引与去重判断。
  - unique block 写入和恢复。
  - 热 digest 加载到 FPGA。
  - GPU/Host/Cold 调度模拟。

- GPU 当前负责：
  - 在模拟器中作为热 KV block 的驻留层。
  - `torch-cuda` 后端可以真实分配显存。
  - 但尚未与真实 attention kernel 或 PagedAttention block table 打通。

### 真实系统中 FPGA 的作用

- FPGA 不负责访问已经登记好的 KVCache。
- FPGA 主要在“需要根据 KV page 内容生成 digest”的时候参与：

```text
新产生的 KV page bytes
  -> FPGA 计算 digest
  -> FPGA 查询 hot table 或 hot mapping table
  -> CPU/runtime 根据 digest 判断是否复用已有 unique block
```

- 如果某个 logical page 已经登记过，系统已经知道：

```text
logical page -> unique_block_id -> physical GPU block id / Host / SSD location
```

则后续访问该 page 时通常不需要 FPGA。

- 因此真实系统应区分两条路径：

```text
写入/识别路径：
KV page bytes -> FPGA hash -> digest -> 去重/复用

读取/访问路径：
logical page id -> block mapping -> GPU/CPU/SSD location -> 读取或取回
```

### 关于 FPGA 热表的进一步设计

- 当前 FPGA hot table 是：

```text
digest -> HOT_HIT/MISS
```

- 可以升级为 hot mapping table：

```text
digest -> block_id / physical_block_id / cache_slot
```

- 推荐返回的是 runtime 能理解的 block handle，而不是 GPU 裸物理地址。
- 原因：
  - GPU 显存地址由 CUDA/LLM runtime 管理。
  - 地址可能迁移、释放、重用。
  - FPGA 通过普通以太网 UDP 不能安全直接管理 GPU 显存。

- 推荐表项结构：

```text
valid
digest[255:0]
block_id[31:0]
location[1:0]       // GPU_PAGE / HOST_CACHE / COLD_FILE，可选
epoch/version[15:0] // 防止旧映射失效，可选
```

- 这样 FPGA 角色可从“热度过滤器”升级为：

```text
FPGA-resident hot mapping table:
content fingerprint -> cached KV block handle
```

### KVCache 是否存在硬盘上

- 正常大模型推理中，KVCache 通常存在 GPU 显存。
- 显存压力下可能 offload 到 CPU 内存。
- 只有长上下文、极冷缓存、跨会话持久化等特殊场景，才可能落到 SSD/硬盘。
- 当前项目中的 `COLD_FILE = chunks/<digest>.bin` 主要是离线验证和原型实验使用，不代表普通推理系统默认把 KVCache 存硬盘。

真实系统更合理的层次应表述为：

```text
GPU_PAGE   = 当前 attention kernel 可直接读取的 KV block
HOST_CACHE = CPU 内存中的 offload KV block
COLD_FILE  = SSD/硬盘上的持久化或极冷备份，可选
```

### 项目与 KVCache 压缩的关系

- 当前项目可以归类为 KVCache 压缩，但更准确地说是：

```text
KVCache 内容寻址去重
content-addressed KVCache deduplication
```

- 它不是传统压缩：
  - 不是对每个 KV page 做编码压缩。
  - 不是 LZ4/Zstd/量化/低秩压缩。

- 它的压缩收益来自重复 KV page：

```text
多个 logical KV page 内容相同
  -> 只保存一个 unique KV block
  -> 多个 logical page 引用同一个 unique block
```

- 如果 KV page 之间没有重复，该方法不会显著减少数据量。

### 真实运行时加/不加本项目的访问区别

不加本项目：

```text
request A page 0 -> GPU block 0
request B page 0 -> GPU block 2
```

即使两个 page 内容相同，通常也各占一份显存。

加上本项目后，目标设计是：

```text
request A page 0 -> digest X -> unique_block X -> GPU block 0
request B page 0 -> digest X -> unique_block X -> GPU block 0
```

即多个 logical KV page 可以共享同一个 physical GPU block。

核心变化：

```text
不加项目：
logical page -> physical GPU block

加上项目：
logical page -> unique_block_id -> physical GPU block
```

代价是需要维护：

```text
digest 索引
logical page -> unique block 映射
unique block -> GPU/CPU/SSD location
引用计数
生命周期和一致性
```

收益是：

```text
减少 GPU 显存占用
减少重复 KV block 的写入和搬运
支持跨 request / shared prefix 的 KVCache 复用
```

### 关于“一个 page 存很多请求”的澄清

- 当前设计不是把多个 request 的数据混放进同一个 page。
- 正确理解是：

```text
一个 logical KV page 属于一个 request
多个 logical KV page 可以因为内容相同而引用同一个 unique KV block
```

- 示例：

```text
request A page 0 -> digest X -> unique_block 1
request B page 0 -> digest X -> unique_block 1
request C page 0 -> digest Y -> unique_block 2
```

- 访问 request A 时，只查 request A 自己的 logical page 映射：

```text
request A, layer, K/V, head_group, token range
  -> logical_page_A0
  -> unique_block 1
  -> physical GPU block 1
```

- request B 也可以通过自己的映射指向同一个 physical block。
- 这不表示 physical page 内部混有多个 request 的数据，而是多个 request 的 page 引用同一份物理内容。

### 当前设计的准确总结

当前项目已经有：

```text
logical KV page -> digest -> unique KV block
```

当前主要落在：

```text
sqlite 元数据 + chunks/<digest>.bin 存储 + GPU/Host/Cold 模拟
```

还没有真正落到：

```text
unique KV block -> vLLM/PagedAttention physical GPU block
```

一句话总结：

```text
当前项目是“离线 KV page 内容寻址去重 + unique block 存储 + GPU/Host/Cold 调度模拟”。
下一步若要成为真实运行时系统，需要把 digest/unique_block 映射接入真实 LLM runtime 的 KV block table。
```

### 后续建议

- 论文/汇报中应避免说“KVCache 默认存硬盘”。
- 更建议描述为：

```text
KVCache 默认驻留 GPU，显存压力下可 offload 到 CPU，进一步可选落盘持久化。
```

- 后续工程重点应从硬盘去重转向真实 runtime 显存共享：

```text
digest -> unique_block_id -> physical GPU block id
```

- FPGA 热表可以进一步升级为：

```text
digest -> cached KV block handle
```

以增强 FPGA 在真实运行时路径中的价值。
