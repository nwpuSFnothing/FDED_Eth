# 2026-04-29 FDED_host 工程构建记录

## 1. 背景

本次对话的核心目标是：

- 判断 host 侧实现 CDC 切块是否可行
- 明确当前 FPGA 端是否已满足 host+FPGA 协同方案需求
- 在 `FDED_host` 目录下搭建新的 Python / PyCharm 工程
- 采用 `sqlite3` 作为本地指纹表
- 明确 CDC 直接使用 `fastcdc` Python 包

## 2. 关键结论

### 2.1 host 侧做 CDC 是可行的

当前系统更适合采用如下分工：

- Host：做 CDC 切块
- FPGA：接收 UDP 请求并计算 SHA-256
- Host：接收 digest 后做精确查重与索引维护

原因是：

- 当前 FPGA 工程本质上是 `UDP -> SHA-256 -> UDP reply` 哈希加速器
- 现阶段还不是完整的“边接收边切块边哈希”的流式去重架构

### 2.2 当前 FPGA 已支持多块 SHA-256，但仍存在系统级长度上限

本次对话中特别澄清了两个不同层面的概念：

- `多块 SHA-256`：指 FPGA 端已经可以对超过单个 `64B` block 的输入做正确 SHA-256 计算
- `单次请求长度上限`：指当前网络链路、UDP 报文组织方式、接收 RAM 和未做 IP 分片重组等条件下，单个请求能稳定传输并处理的上限

因此当前应准确理解为：

- FPGA 的 SHA-256 计算路径已经支持多块输入
- 但整套系统仍受单个 UDP 请求长度和接收缓存组织方式约束
- 当前稳定实测的真实 hash 数据上限仍按 `1468B` 理解更合理

### 2.3 `ultraembedded/core_usb_cdc` 与本工程需求无关

对话中还澄清了：

- 本工程语境中的 `CDC` 指 `Content-Defined Chunking`
- 不是 USB 的 `CDC-ACM`

因此 `core_usb_cdc` 这种 USB CDC 设备核并不适合当前以太网去重/哈希链路。

## 3. Host 工程设计决策

本次最终确定：

- 在 `FDED_host` 下构建独立 Python 工程
- 本地指纹表采用 `sqlite3`
- CDC 直接依赖 `fastcdc` Python 包
- FPGA 通信采用现有已验证协议：
  - 请求：`4B seq_id + payload`
  - 响应：`4B seq_id + 32B digest`

本地指纹表设计为两张表：

- `fingerprints`
  - 主键：`digest`
  - 保存 `chunk_length`、首次出现路径、首次偏移、引用计数
- `chunk_events`
  - 保存每个 chunk 的处理事件，包括：
  - 源文件
  - chunk 序号
  - 偏移
  - 长度
  - digest
  - 是否重复

## 5. 当前代码实现内容

### 5.1 `chunker.py`

实现了两种切块方式：

- `fixed_size_chunks`
- `fastcdc_chunks`

其中：

- `fastcdc_chunks` 已改为顶层强制导入：
  - `from fastcdc import fastcdc`
- 并直接调用 `fastcdc(file_path, min_size, avg_size, max_size)`

这意味着：

- `fastcdc` 已经被视为硬依赖
- 若环境中未安装该包，程序启动时会直接失败

### 5.2 `fpga_client.py`

实现了 host 侧的 FPGA UDP 客户端逻辑，用于：

- 创建 UDP socket
- 发送 `4B seq_id + payload`
- 接收 `4B seq_id + 32B digest`
- 校验 reply 长度和 `seq_id`

### 5.3 `fingerprint_db.py`

实现了基于 `sqlite3` 的本地指纹库，用于：

- 初始化数据库表
- 记录新的 digest
- 更新重复 digest 的引用计数
- 记录 chunk 处理事件

### 5.4 `pipeline.py`

实现了 host 侧的处理管线：

- 读取输入文件
- 按 `fixed` 或 `cdc` 模式切块
- 将每个 chunk 发给 FPGA 计算 SHA-256
- 将返回 digest 写入 sqlite3 指纹表
- 统计：
  - 总 chunk 数
  - 唯一 chunk 数
  - 重复 chunk 数

### 5.5 `main.py`

作为 CLI 入口，当前支持：

- `process-file`

### 11.2 实测结果

用户反馈终端结果为：

- `bytes = 116606704`
- `chunks = 123499`
- `unique = 123499`
- `duplicate = 0`
- `dup_ratio = 0.0000`
- `elapsed_s = 37.4543`

### 11.3 对结果的判断

对上述结果的分析结论为：

- 结果本身是正常的
- 说明 FPGA 哈希链路和本地 `hashlib.sha256()` 校验一致
- `verify_local` 未报错，证明当前 `Host -> FPGA -> Host` SHA-256 功能链已跑通

同时，`duplicate = 0` 也并不异常，原因是：

- 被测试对象是 `.tar.xz` 压缩包
- 压缩后的单文件通常会将原始重复模式打散
- 因此单次扫描时，块内重复和块间重复都可能非常低

所以该结果应解释为：

- 功能验证成功
- 数据集本身不利于展示去重率收益
