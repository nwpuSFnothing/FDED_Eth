## 3. 当前 FPGA 热指纹表更新时间点

当前 FPGA 热指纹表不会在普通 hash 请求中自动插入新 digest。

它只在 Host 发送 control packet 时更新：

- `clear-hot-table`：清空 FPGA 热表 valid 位。
- `write-hot-digest`：写入指定 slot。
- `load-hot-table`：从 sqlite 读取 top-N digest 后逐项写入 FPGA。
- `process-file/process-dir --load-hot-table`：处理开始前加载一次。
- `--hot-refresh-interval-s N`：处理中每隔约 `N` 秒由 Host 主动刷新一次。

也就是说，运行中 sqlite 的 `ref_count` 改变后，FPGA 热表只有在下一次 Host 主动加载或周期刷新时才会同步。

## 4. 测试现象

执行：

```powershell
python -B main.py process-file sample.bin --load-hot-table --hot-limit 512 --hot-refresh-interval-s 10 --verify-hot-hit --verify-local --print-chunks
```

第一次报错原因是当前目录没有 `sample.bin`，不是协议或热表逻辑错误。

随后运行中曾看到类似统计：

```text
fpga_hot_hits 0
host_lookups  26
lookup_saved  0
hot_hit_ratio 0.0000
hot_loaded    512
hot_refreshes 1
```

含义：

- FPGA 热表加载动作执行过。
- 本次处理的 chunk digest 没有命中已加载的热表。
- 运行时间短于 10 秒，因此只发生开始时的 1 次加载，没有触发周期刷新。

后续执行：

```powershell
python -B main.py process-dir data --recursive --load-hot-table --hot-limit 512 --hot-refresh-interval-s 10 --verify-hot-hit --verify-local
```

实际处理的是：

```text
FDED_host/data/fingerprints.db
```

原因是输入目录 `data` 与默认 sqlite 数据库目录相同，程序把数据库文件本身当作测试数据处理。后续测试应将输入数据目录和数据库路径分开，例如把数据库放到 `Result/hot_test.db`。

## 7. 文件恢复逻辑补充

后续又把文件恢复能力正式加入 Host 工程，不作为临时验证脚本处理。

设计结论：

- FPGA 端不需要修改。
- FPGA 仍只负责 `chunk_data -> SHA-256 digest -> HOT_HIT/MISS`。
- 文件恢复完全由 Host 根据 sqlite 元数据和本地 chunk 存储完成。

Host 侧新增数据库元数据：

- `file_runs`
  - 每次处理一个文件生成一条 run 记录。
  - 保存 `source_path`、`total_bytes`、`chunk_count`、分块参数和创建时间。
- `chunk_events.run_id`
  - 每个 chunk event 关联到所属 file run。
  - 恢复时按 `run_id + chunk_index` 排序拼接。

Host CLI 新增：

```powershell
python -B main.py list-runs --db-path Result\hot_test.db --limit 20
python -B main.py restore-file --db-path Result\hot_test.db --run-id 1 --write-dir F:\Write_in --output-file restored\a.bin
```

也支持按源文件路径恢复最近一次记录：

```powershell
python -B main.py restore-file --db-path Result\hot_test.db --source-path C:\path\to\a.bin --write-dir F:\Write_in --output-file restored\a.bin
```

恢复流程：

```text
file_runs/run_id
  -> chunk_events 按 chunk_index 排序
  -> 读取 write_dir/chunks/<digest>.bin
  -> 顺序写入 output_file
  -> 校验恢复字节数与 file_runs.total_bytes 一致
```

注意事项：

- 恢复依赖处理时写出的唯一 chunk 文件，不能删除 `write_dir/chunks`。
- 如果处理时使用了非默认 `--write-dir`，恢复时也必须传入相同目录。
- 对于同一源文件多次处理的情况，应优先用 `run_id` 恢复，避免歧义。
